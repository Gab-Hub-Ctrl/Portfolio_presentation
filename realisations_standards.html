Dashboard_IT repose sur un corpus de 31 standards rédigés au format professionnel. Chaque standard définit une règle applicable immédiatement par un agent IA ou un développeur humain. L'objectif : ne jamais improviser les décisions récurrentes.

Les standards couvrent l'ensemble du cycle de vie du projet, de la qualité du code aux conventions d'interface, en passant par la sécurité et la gestion des données.

Le guide agent IA intégré à chaque standard permet à un LLM de produire du code conforme sans instructions supplémentaires. La checklist de conformité permet de valider rapidement la conformité d'un fichier existant.

Voici les sections les plus représentatives de STD-CPY-001, le standard de code Python du projet. Il illustre la structure et la profondeur de l'ensemble des 31 standards rédigés.

Référence STD-CPY-001
Version 1.0.0
Date 2025-12-24
Catégorie Code / Python
Auteur Gabriel ROULON
OBLIGATOIRE

1. Portée et objectifs

S'applique à tout fichier .py dans Dashboard_IT/Backend, Dashboard_IT/Tools et Dashboard_IT/Tests. Objectifs : cohérence, lisibilité, maintenabilité, et compatibilité native avec les agents IA du projet.

2. Guide Agent IA (9 étapes)

01 Lire le fichier complet avant toute modification
02 Identifier la structure : module standalone, classe, ou fonction
03 Appliquer les conventions de nommage : snake_case pour fonctions/variables, PascalCase pour les classes, UPPER_SNAKE_CASE pour les constantes
04 Vérifier les type hints sur toutes les signatures publiques
05 Vérifier la documentation : docstrings Google style sur toutes les fonctions et classes publiques
06 Appliquer la gestion d'erreurs : try/except ciblé, exceptions custom, logging adapté
07 Valider : black (formatage), flake8 (lint), mypy (typage) avant finalisation
08 Mettre à jour CHANGELOG si modification majeure ou breaking change
09 Exécuter la checklist de conformité (section 23 du standard)

3. Prompt type condensé

Applique STD-CPY-001 v1.0.0 au fichier Python fourni.
Vérifie : nommage (snake_case / PascalCase / UPPER_SNAKE_CASE),
type hints complets sur toutes les signatures publiques,
docstrings Google style, gestion d'erreurs (try/except ciblé, exceptions custom, logging),
conformité black / flake8 / mypy.
Score cible : ScoreGlobal ≥ 90/100.
Formule : (Structure×0.25) + (Nommage×0.20) + (Portabilité×0.15) + (TypeHints×0.20) + (Documentation×0.20)

23. Checklist de validation

  • Structure fichier conforme : imports ordonnés (stdlib / tiers / local), module docstring présent, __all__ défini si module public
  • Nommage respecté : snake_case fonctions/variables, PascalCase classes, UPPER_SNAKE_CASE constantes
  • Type hints présents sur toutes les signatures publiques (paramètres et valeur de retour)
  • Docstrings Google style sur toutes les fonctions, méthodes et classes publiques
  • Gestion d'erreurs : try/except ciblé (pas de except Exception nu), exceptions custom héritant de la hiérarchie projet, logging adapté
  • Tests unitaires présents, couverture cible ≥ 80 %
  • Formatage black : aucune erreur ni différence
  • Linting flake8 : aucun avertissement bloquant (E/F)
  • Typage mypy : aucune erreur de type
  • Score global calculé ≥ 90/100 selon la formule de pondération

Rédiger 31 standards n'est pas une fin en soi. C'est la traduction d'une conviction : un projet IA sans règles écrites est un projet incontrôlable.

Chaque standard est une décision prise une seule fois, documentée, puis appliquée systématiquement par les agents et les humains. C'est la base d'une gouvernance IA sérieuse.

Ces standards ont été rédigés progressivement, au fil des besoins réels du projet. Ils ne sont pas théoriques : chaque règle a été nécessitée par une situation concrète rencontrée lors du développement de Dashboard_IT.
